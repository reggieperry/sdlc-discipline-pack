# sdlc-discipline pack: v1.3 vs v2.0 comparison and ship decision

This document captures the empirical comparison between the v1.3 named-session architecture and the v2.0 polecat-pool architecture, validated by three runs: two v1.3 baseline chains, one v2.0 single-pilot replay, and one v2.0 ten-story concurrent batch.

Source data lives in:

- `v1.3-baseline.json` — per-phase metrics for the two completed v1.3 chain runs (csv2json/cs-4b2q, gh-stats/gs-id4)
- `v2.0a-stall-record.json` — the gs-hdpw stall record from the interim v2.0a (kept for posterity; v2.0 fixes the routing bug)
- `v2.0-results.json` — Phase 3 single-pilot replay (gs-uud)
- `v2.0-phase4-concurrent-results.json` — Phase 4 ten-story concurrent batch
- `stories/` — verbatim story bodies for each run, replayable

## Architecture summary

| | v1.3 | v2.0 |
|---|---|---|
| Pack agents | 5 `[[named_session]]` (planner, implementor, tester, reviewer, documenter) | 5 pools (worker, tester, reviewer, documenter, finalizer); zero named sessions |
| Concurrency model | One canonical instance per template per rig; serial within a rig, parallel across rigs | Pool reconciler scales each template up to its `max_active_sessions`; parallel within a rig and across rigs |
| Workaround required | Yes — gascity#1893 alive-session reset at kickoff | No |
| Routing convention | Mixed: `--assignee` and `gc.routed_to` both used | `gc.routed_to` only at every pool transition; `--assignee` reserved for pool-internal claim after spawn |
| Per-rig max concurrent sessions | 5 (one of each named) | 15 (5 worker + 3 tester + 3 reviewer + 2 documenter + 2 finalizer) |
| Merge gate | Documenter handled push + PR + auto-merge inline | Finalizer pool owns merge gate; documenter writes feature docs only |
| Worker scope | Planner wrote plan; implementor did R/G/R; tester ran pytest; each in a separate session | Worker walks load-context → plan → workspace-setup → implement → self-audit → submit in one session; tester runs pytest + ruff + mypy in a fresh session |

## Single-story comparison

Same story body (`gh-stats: list-prs --repo OWNER/REPO`), filed verbatim against v1.3 (gs-id4) and v2.0 (gs-uud).

| Metric | v1.3 (gs-id4) | v2.0 (gs-uud) | Delta |
|---|---|---|---|
| Total wall clock | 25m 36s | 46m 19s | +81% |
| Phase work | 17m 33s | 34m 6s | +94% |
| Inter-phase gaps | 8m 03s | 12m 11s | +51% |
| Plan + build (planner+implementor in v1.3; worker in v2.0) | 10m 32s | 22m 17s | +112% |
| Tester | 21s | 1m 26s | +310% (now runs ruff+mypy) |
| Reviewer | 3m 24s | 5m 41s | +67% |
| Documenter | 3m 16s | 3m 51s | +18% |
| Finalizer | n/a | 51s | new |
| Commits on feature branch | 1-3 (squashed) | 16 (Two-Hats split) | — |
| Tests added | 12 | 15 | +3 |

Per-story v2.0 is **3.2× slower in worker time** and **1.8× slower end-to-end** than v1.3. The ratio reflects two overhead sources: the heavier discipline rules produce more commits per story (Two-Hats split, R/G/R per behavior, self-audit walks), and the additional pool handoffs (worker→tester→reviewer→documenter→finalizer is five handoffs vs v1.3's four) add per-handoff spawn latency.

## Concurrent-batch comparison

v1.3 cannot run multiple stories concurrently within a rig; serial within each rig is the architectural ceiling. The fairest v1.3 hypothetical for ten stories spread across two rigs:

> Each rig serializes its five stories: 5 × ~25m = ~2h 5m per rig. Both rigs run in parallel. Total batch wall clock: ~2h 5m.

v2.0 actual Phase 4 batch (5 csv2json + 5 gh-stats, slung simultaneously):

> Total batch wall clock: **2h 27m 42s**. Excluding the 35-50 min stall on two gh-stats stories from the no-remote finalizer bug (fixed in v2.0.1): ~**1h 50m**.

v2.0 advantage at the 10-story scale: **~1.13× speedup** in clean conditions. Modest. The advantage grows with batch size — at 50 stories per rig, v1.3 takes 50 × 25m = 20+ hours per rig serial; v2.0 runs the worker pool at max=5 and finishes in roughly the time of 10 sequential stories × pool wait, so an order of magnitude faster.

The break-even where v2.0 wins on wall clock is somewhere around 5-7 stories per rig. Below that, v1.3's lighter per-story work makes it faster overall; above that, v1.3's serialization dominates.

## Quality comparison

Across the runs:

- **All v1.3 chains** produced review-pass, test-green output. cs-4b2q failed glance R4 on the size cap and was human-merged; gs-id4 was branch-only (no remote).
- **All v2.0 chains** produced review-pass, test-green output. cs-4b2q's analog stories on csv2json (Phase 4) all failed glance R4 and were left for human review; all gh-stats stories landed cleanly on local branches.
- **v2.0 produced more granular commits** (12-16 per story vs 1-3 per story for v1.3). This is the Two-Hats discipline working as designed: each commit is a single atomic change with a single hat (`feat:`, `test:`, `chore:`, `refactor:`, `docs:`).
- **Reviewer findings** were comparable across versions: zero blockers in any of the runs we observed; one or two `nit`-level findings per story.
- **Generated feature docs** were factually grounded in both versions — no hallucination of file paths or line numbers.

The discipline rules don't make v2.0's code meaningfully *better* per acceptance-criterion-passed; they make it more *uniformly* shaped — every story produces a similar commit history, similar test count, similar doc structure. That uniformity is the load-bearing value, particularly for unattended batch runs where reviewer attention is scarce.

## Bugs and follow-ups

Surfaced during v2.0 validation, all tracked on the project's fit-and-finish list:

1. **No-remote finalizer stall** — fixed in v2.0.1 (commit 0b415ab, tagged 2026-05-09). Prompt now detects missing origin and closes as `branch_ready_no_pr` without an interactive chooser.
2. **Glance failure-reason metadata missing** — low priority; the rubric output is in the PR comment, just not lifted into bead metadata.
3. **Gas City supervisor / Dolt CPU overhead** — supervisor reconciler polls at ~1-2 qps; Dolt sits at 70-100% CPU even when idle; potentially upstream issue worth filing once we have repro evidence.
4. **Pack v2.1 lighter-discipline** — empirical evidence from gs-uud commits shows Two-Hats commit splitting and R/G/R per cycle are partially ceremonial. Estimated 25-35% per-story saving from a lighter shape that keeps the load-bearing pieces (plan-first, mocks-as-peers, behavior-named tests, sensitive-files declaration).
5. **Kickoff phase collapse** — the kickoff step runs as a fresh Claude Code session just to execute four `bd` commands. Replace with synchronous bash inline in `gc sling`. Saves 1-2 min per story and removes the synchronized 10-session boot pulse from concurrent batches (~3.5 GiB peak RAM during the boot pulse on Phase 4).
6. **Prompt-clarity discipline audit** — `--dangerously-skip-permissions` doesn't bypass the model's own clarification questions. Every prompt's deterministic paths must be exhaustive or unattended chains stall on choosers. Audit all five v2.0 prompts for branches that could surface a chooser under realistic edge cases.

## Decision

**Ship v2.0** (released as v2.0.1 with the finalizer fix).

The case for shipping:

- The architectural deficit v1.3 had was a hard concurrency ceiling. v2.0 removes it. The break-even for v2.0 wall-clock advantage is about 5-7 stories per rig; above that, v1.3 doesn't compete.
- The single bug v2.0 introduced that blocked shipping (no-remote finalizer stall) is fixed and tagged. The other bug (glance failure-reason capture) is cosmetic.
- The remote-install path is validated end-to-end. Adopting v2.0 from a fresh Gas City workspace works as documented in the README's seven-step install guide.
- The fit-and-finish items address known overhead and ship as v2.1; they don't gate v2.0.

The case against shipping (and how it's addressed):

- Per-story latency is 3.2× v1.3. **For single-story iterative work on a laptop, v1.3 is faster.** Address by running v2.0 with the lighter-discipline shape (v2.1) once it lands, or by accepting that v2.0 is the batch-throughput tool while single-story iteration uses lighter ad-hoc workflows.
- Supervisor overhead (Dolt CPU, kickoff pulse) is meaningful on memory-constrained machines. **Documented; tracked under fit-and-finish.** Workstation (T7920) handles it cleanly; 16 GB laptops will benefit materially from the kickoff-phase collapse.
- Heavy discipline produces more commits per story than necessary. **Tracked under fit-and-finish #2.** The lighter shape (v2.1) keeps the load-bearing pieces and drops the ceremony.
